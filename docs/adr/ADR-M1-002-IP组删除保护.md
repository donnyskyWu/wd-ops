## 1. 问题

IP 组管理（FR-M1-001）中，是否允许删除 IP 组？
- 简单删除：直接 DELETE
- 软删除：标记 `deleted=1`
- 保护删除：存在子数据时禁止删除

---

## 2. 决策

**采用"保护删除 + 软删除"组合**：

1. **保护删除**：当 IP 组下存在成员/账号/主播/历史数据时，**禁止物理删除**
2. **软删除**：通过 `status=0`（停用）实现"逻辑删除"
3. **物理删除**：仅在**测试环境**或**新建 24h 内无任何数据**时允许

---

## 3. 理由

- **避免误删**：IP 组下通常有关联账号、成员、主播、操作历史，物理删除会导致历史数据无法回溯
- **审计需要**：删除会破坏审计日志关联（"谁在什么时间操作了这个 IP 组"）
- **业务连续性**：IP 组"停用"已经能满足"业务下线"需求
- **法规要求**：金融/数据相关行业要求保留历史数据 N 年

---

## 4. 规则

| 状态 | 物理删除 | 停用（软删） | 说明 |
|------|----------|--------------|------|
| IP 组下存在成员 | ❌ | ✅ | 提示 1005 "存在数据，禁止删除" |
| IP 组下存在账号 | ❌ | ✅ | 同上 |
| IP 组下存在主播 | ❌ | ✅ | 同上 |
| IP 组下存在历史数据（作品/订单） | ❌ | ✅ | 同上 |
| IP 组下完全无数据（新建 24h 内） | ✅（仅管理） | ✅ | 仅管理员可操作 |
| 测试环境 | ✅ | ✅ | 不受保护 |

---

## 5. 业务校验

后端校验逻辑：

```python
def can_delete_ip_group(group_id):
    group = db.get_ip_group(group_id)
    if not group:
        return False, "IP 组不存在"
    
    # 检查子数据
    if db.count_ip_group_members(group_id) > 0:
        return False, "存在成员，禁止删除"
    if db.count_ip_group_accounts(group_id) > 0:
        return False, "存在账号，禁止删除"
    if db.count_ip_group_anchors(group_id) > 0:
        return False, "存在主播，禁止删除"
    if db.count_historical_data(group_id) > 0:
        return False, "存在历史数据，禁止删除"
    
    # 检查存在时间
    if group.created_at < now() - 24h and not is_admin(user):
        return False, "新建超过 24h 需管理员权限"
    
    return True, None
```

## 6. UI 表现

| 情况 | UI |
|------|------|
| 存在数据 | 删除按钮置灰 + tooltip "该 IP 组下存在数据，禁止删除" |
| 无数据 | 删除按钮可点击 + 二次确认弹窗 |
| 24h 内新建 | 删除按钮可点击 |
| 24h 后 + 无数据 | 删除按钮可点击，但需管理员权限 |

---

## 7. 替代方案

### 方案 A：直接物理删除

- 优点：简单
- 缺点：风险高，破坏审计
- 不采用 ❌

### 方案 B：纯软删除（无 status 字段）

- 优点：实现简单
- 缺点：无法区分"已删除"和"停用"
- 不采用 ❌

### 方案 D：保留 30 天后自动清理

- 优点：节省空间
- 缺点：与法规要求可能冲突
- 不采用（一期）❌

---

## 8. 影响

- 操作日志：所有删除尝试（成功+失败）都记录
- 数据恢复：仅支持"停用→启用"恢复，不支持"删除后恢复"
- 管理员权限：物理删除需管理员；普通运营者只能"停用"
