# ADR-017：M2 可配置二级内容审核与 IP 组审核范围

| 字段 | 值 |
|------|---|
| 编号 | ADR-017 |
| 标题 | 系统参数驱动的二级审核 + IP 组长范围审核 |
| 状态 | ✅ Accepted（2026-06-13 代码已落地） |
| 日期 | 2026-06-13 |
| 决策人 | 产品（近期实现会话对齐） |
| 关联 PRD | `PRD-M2-内容生产.md` · FR-M2-003 |
| 关联 API | `API-M2-内容生产.md` §3.4~3.5 · `GET /content/review-config` |
| 关联 M9 | `PRD-M9-系统管理.md` · FR-M9-004 系统参数 · Tab「内容审核」 |

---

## 1. 背景

原 PRD 描述固定「三级审核（初审→复审→终审）」。近期实现改为：

1. 审核级数与审核角色由 **M9 系统参数** 配置（可开/关一级、二级）
2. 一级审核角色默认为 `OPS_LEADER` 时，审核范围限定为 **内容所属 IP 组的组长**，而非全租户该角色用户
3. 独立创作与任务驱动创作统一走「**提交审核**」，不再使用「确认」进入发布流（任务完成门禁仍用 `COMPLETED`，见 ADR-016）

---

## 2. 决策

### 2.1 系统参数键（租户级）

| 参数键 | 类型 | 默认 | 说明 |
|--------|------|------|------|
| `content.review.level1.enabled` | boolean | `true` | 是否启用一级审核 |
| `content.review.level2.enabled` | boolean | `true` | 是否启用二级审核 |
| `content.review.level1.role` | string（角色 code） | `OPS_LEADER` | 一级审核角色；值为 `OPS_LEADER` 时走 IP 组长范围 |
| `content.review.level2.role` | string | `DEPT_HEAD` | 二级审核角色 |

- 参数管理 Tab「内容审核」中，`level1.role` / `level2.role` 使用 **角色下拉**（非手输）。
- 至少保留一级或二级之一开启；**两者均关闭**时，`submit-review` 直接进入 `PUBLISHED`（跳过审核）。

### 2.2 状态流转（可配置）

| 配置 | 提交后状态 | 一级通过后 | 二级通过后 |
|------|-----------|-----------|-----------|
| L1✅ L2✅ | `PENDING_FIRST_REVIEW` | `PENDING_SECOND_REVIEW` | `PUBLISHED` |
| L1✅ L2❌ | `PENDING_FIRST_REVIEW` | `PUBLISHED` | — |
| L1❌ L2✅ | `PENDING_SECOND_REVIEW` | — | `PUBLISHED` |
| L1❌ L2❌ | `PUBLISHED` | — | — |

- 字典仍保留 `PENDING_FINAL_REVIEW` / `FINAL_REVIEW` 以兼容历史数据；**默认配置不使用终审**。
- 驳回：任一环节驳回 → `REJECTED`，流程结束。

### 2.3 审核权限

| 阶段 | 角色配置 | 权限规则 |
|------|---------|---------|
| 一级 `FIRST_REVIEW` | `OPS_LEADER` | 仅 `oa_ip_group.leader_user_id = 当前用户` 且内容 `ip_group_id` 匹配 |
| 一级 `FIRST_REVIEW` | 其他角色 code | 拥有该 `sys_role.code` 的用户可审全部待一级内容 |
| 二级 `SECOND_REVIEW` | 配置的 `level2.role` | 拥有该角色的用户可审全部待二级内容 |

- 审核列表 API 按 `hasLevel1ListAccess` / `hasLevel2ListAccess` 过滤。
- 审核页 Tab：**一级审核** / **二级审核**（随 enabled 动态显示）。

### 2.4 审核流程展示

- 内容只读查看（列表「查看」、审核抽屉、待审只读）展示 **审核流程** `el-steps`。
- 每步显示：**角色标签 + 可审核用户姓名**（如「IP组长：张三」）。

### 2.5 与任务场景关系

- 任务驱动创作：`保存` → `DRAFT`；**提交审核** → 进入 §2.2 可配置审核流（不再推荐「确认」按钮；遗留 `COMPLETED` 数据可再提交审核）。
- 任务节点完成门禁（ADR-016）：仍要求关联内容存在；**不再强制** `status=COMPLETED`，改为须已 **提交审核**（`PENDING_*`）或审核通过/发布（实现以 `submitReview` 后状态为准，见 BLK-M2-006 部分闭合）。

## 3. 后果

| 层 | 变更 |
|----|------|
| M9 | 系统参数 Tab + Seed 四条审核参数；角色下拉编辑 |

| M2 UX | 审核页双 Tab；内容编辑/列表「提交审核」；只读审核流程 steps |

| BLK-M2-006 | 部分闭合：任务场景与审核流已衔接；`COMPLETED` 遗留兼容保留 |
