## 1. 背景

v9.1 之前的架构在产品逻辑架构中包含以下中间件：
- **Redis**：用作缓存
- **RabbitMQ**：消息队列
- **MinIO**：对象存储
- **XXL-JOB**：分布式任务调度

团队评估后认为，这些中间件在当前业务规模和并发量下**并非必需**，反而增加了：
- 部署复杂度（多组件运维）
- 团队学习成本
- 故障点（中间件宕机会级联影响业务）
- 资源占用（内存、磁盘）

---

## 2. 决策

| 旧方案 | 新方案 | 理由 |
|--------|--------|------|
| Redis 缓存 | MySQL + 本地 `ConcurrentHashMap` | 业务并发量适中，MySQL 索引已能支撑；热数据用 JVM 内 Map 缓存 |
| RabbitMQ 消息队列 | Spring `@Async` + 本地任务表 | 异步场景集中在数据采集、报表导出；分布式消息能力用 DB 任务表 + 定时轮询替代 |
| MinIO 对象存储 | 本地文件 + Nginx 静态代理 | 导出文件、附件大小可控（< 50MB/文件），不需要分布式对象存储 |
| XXL-JOB 分布式调度 | Spring `@Scheduled` + 数据库任务表 | 单体部署足够，调度逻辑可观测可重试；分布式调度功能本期不需要 |

---

## 3. 影响范围

### 3.1 受益

- **部署简化**：从"8 个组件"（MySQL + Redis + MQ + MinIO + XXL-JOB + Nginx + App + 监控）减为"4 个"（MySQL + Nginx + App + 监控）
- **运维成本下降**：不需要 Redis/MQ/MinIO 集群维护
- **故障点减少**：故障爆炸半径从"中间件 + 业务"变为"仅业务"

### 3.2 代价

- **横向扩展能力受限**：单体架构不利于水平扩展，但当前业务量（中等并发，1k~10k DAU）足够
- **跨实例消息**：通过数据库任务表 + `SELECT ... FOR UPDATE SKIP LOCKED` 实现分布式任务领取
- **大文件存储**：单文件限制 50MB，超过需走对象存储（一期不涉及）

### 3.3 未来可恢复

如果业务规模增长到需要这些中间件时，可逐项引入：
- 先引入 Redis（缓存）
- 再引入 RabbitMQ（解耦）
- 必要时引入 XXL-JOB（分布式调度）
- 最后才考虑 MinIO（大规模文件存储）

每次引入需新开 ADR。

---

## 4. 技术约束

- 🔴 **不引入 Redis**：缓存需求用 JVM Map（`Caffeine` 或 `ConcurrentHashMap`）
- 🔴 **不引入 RabbitMQ**：异步用 `@Async` + 任务表
- 🔴 **不引入 MinIO**：文件用本地路径 + Nginx
- 🔴 **不引入 XXL-JOB**：调度用 `@Scheduled` + 任务表

详见 `docs/engineering/TECH-CONSTRAINTS.md § 1.1`

---

## 5. 替代方案对比

### 方案 A：保持原方案

| 维度 | 评价 |
|------|------|
| 性能 | 略好（Redis 缓存） |
| 复杂度 | 高 |
| 资源占用 | 高 |
| 运维成本 | 高 |

### 方案 B：简化（已选 ✅）

| 维度 | 评价 |
|------|------|
| 性能 | 略低但够用 |
| 复杂度 | 低 |
| 资源占用 | 低 |
| 运维成本 | 低 |

### 方案 C：微服务架构

| 维度 | 评价 |
|------|------|
| 性能 | 高 |
| 复杂度 | 极高 |
| 资源占用 | 极高 |
| 运维成本 | 极高 |

> 一期业务规模不需要微服务。

---

## 6. 验证

- 业务量 1k~10k DAU 场景下，新方案无明显性能瓶颈
- 异步任务（数据采集、报表导出）通过 `@Async` + 任务表成功处理 1w+ 任务/日
- 本地文件 + Nginx 支持 5GB/日下载量
