# ADR-045: M10 奥创多账号与设备同步

> **状态**: 已采纳 | **日期**: 2026-06-18
> **决策人**: 产品 + 开发团队
> **关联**: [ADR-014](./ADR-014-M8-配置管理数据模型.md) · [三通道采集计划](../delivery/M10-三通道采集计划.md) · [9446d1be 三通道方案](https://cursor.com) · [145d7c45 奥创 API 65947279 修正](https://cursor.com) · [a1f1ec21 AI 身份关联 P2](https://cursor.com)

## 背景

奥创 OpenAPI 集成需支持：租户级凭证、**多个**奥创客服工作台账号、自动发现个微设备、绑定 OA 个微档案、采集好友与消息，并接入三通道采集架构（Channel-B）。

先前方案（145d7c45）拟在 `oa_aocreate_api` 上增加单列 `aochuang_account_id`；产品确认 **一租户可有多个奥创账号**，故改为子表设计。

## 决策摘要

| 编号 | 问题 | 决策 |
|------|------|------|
| **Q1** | 租户奥创账号数量 | **多账号**：新建 `oa_aocreate_account` 子表；**不在** `oa_aocreate_api` 上放 `aochuang_account_id` |
| **Q2** | 设备匹配策略 | **wechatId/alias 精确自动绑定**；**nickname + avatar 模糊 ≥ 0.85 也自动绑定**（非仅人工队列） |
| **Q3** | 奥创有设备、OA 无档案 | **B**：进「待绑定列表」，运营确认后新建或绑定 |
| **Q4** | 手工 override | **A**：保留高级选项，默认只读；审计 `bind_method=MANUAL` |
| **Q5** | 设备与 OA 行关系 | **B**：`UK (tenant_id, aochuang_wechat_account_id)`，一设备对应一个个微档案行 |

### 已确认前置（沿用）

- **个微配置落点**：`个人账号管理 → 个人微信` Tab（`PersonalAccountManage.vue`），非平台账号管理
- **凭证 SSOT**：M8 `oa_aocreate_api`（apiUrl / appId / appSecret / token），管理员专用
- **三通道架构**（9446d1be）：`AochuangAdapter` 独立，共享 M10 任务壳
- **跨通道身份关联**：AI 头像/昵称增强为 **P2**（a1f1ec21）；本 ADR 仅覆盖奥创设备绑定

## 1. 数据模型

### 1.1 凭证层 — `oa_aocreate_api`（保持，不增 accountId 列）

租户级一条（`UK tenant_id`），仅存 HTTP 鉴权凭证：

| 列 | 说明 |
|----|------|
| `tenant_id` | 租户隔离 |
| `api_url` / `app_id` / `app_secret_encrypted` / `token_encrypted` | AES-256 |
| `status` / `daily_quota` / `current_usage` | 启用与配额 |

> **变更**：撤销「在 `oa_aocreate_api` 增加 `aochuang_account_id`」方案（145d7c45 §3.1）。

### 1.2 奥创账号层 — `oa_aocreate_account`（新建）

一个租户可配置 **多条** 奥创客服/管理工作台账号：

| 列 | 类型 | 约束 | 说明 |
|----|------|------|------|
| `id` | BIGINT | PK | |
| `tenant_id` | BIGINT | NOT NULL | 租户隔离 |
| `aocreate_api_id` | BIGINT | FK → `oa_aocreate_api.id` | 关联租户凭证 |
| `account_name` | VARCHAR(100) | NOT NULL | 运营可读名称（如「华东客服组」） |
| `aochuang_account_id` | VARCHAR(64) | NOT NULL | 奥创 path 参数；`/accounts/{accountId}/wechatAccounts` |
| `status` | VARCHAR(32) | `@InDict dict_common_status` | ENABLED / DISABLED |
| `last_device_sync_at` | TIMESTAMP | NULL | 最近设备列表同步 |
| `conn_status` | VARCHAR(32) | NULL | OK / TOKEN_FAIL / PERMISSION_DENIED |
| 审计列 | | | creator / create_time / updater / update_time / deleted |

**唯一键**：`UK (tenant_id, aochuang_account_id)`

**M8 UI**：个人微信 Tab 在凭证表单下方增加「奥创账号列表」CRUD + 单账号「测试连接 / 同步设备预览」。

### 1.3 个微设备绑定 — `oa_personal_wechat_account` 扩展

| 列 | 说明 |
|----|------|
| `aochuang_wechat_account_id` | 奥创 `wechatAccountId`；同步写入，**默认只读**（Q4） |
| `aochuang_account_ref_id` | FK → `oa_aocreate_account.id`（来源奥创账号） |
| `aochuang_bind_status` | UNBOUND / AUTO / MANUAL / PENDING |
| `aochuang_nickname` / `aochuang_avatar` / `aochuang_is_alive` | 设备快照（匹配与展示） |
| `last_device_sync_at` / `last_friend_sync_at` / `last_message_sync_at` | 同步时间 |
| `collect_status` | 采集状态字典 |

**唯一键（Q5）**：`UK (tenant_id, aochuang_wechat_account_id)` — 非空时生效；一奥创设备仅绑一个个微档案行。

**待绑定队列（Q3）**：同步发现未匹配设备时写入 staging（或 `bind_status=PENDING` + 独立 `oa_aochuang_device_pending` 视图），运营在「同步个微设备」对话框确认「绑定已有 / 新建并绑定」。

### 1.4 采集事实表（沿用 899566a5 / 9446d1be 命名）

| 表 | 用途 |
|----|------|
| `oa_aochuang_friend` | 好友快照 |
| `oa_aochuang_message` | 消息明细 |
| `oa_personal_wechat_daily_stats` | 个微日聚合 |
| `oa_aochuang_sync_cursor` | 瀑布流游标 |

## 3. 设备匹配规则

匹配在 `DeviceSyncService` 执行，按优先级 **短路**；命中即写入 `aochuang_wechat_account_id` 并设 `bind_status=AUTO`。

| 优先级 | 规则 | 条件 | 动作 |
|--------|------|------|------|
| **P0** | wechatId 精确 | `dto.wechatId` = `oa_personal_wechat_account.wechat_id`（忽略大小写） | 自动绑定 |
| **P1** | alias 精确 | `dto.alias` 非空且 = `wechat_id` | 自动绑定 |
| **P2** | nickname + avatar 模糊 | 见 §3.1 | **自动绑定**（Q2 确认） |
| **P3** | 无匹配 | 奥创有设备、OA 无对应行 | **待绑定列表**（Q3=B） |
| **兜底** | 手工 override | 运营在高级选项修改 ID | `bind_status=MANUAL`，审计 `linked_by` |

### 3.1 模糊匹配阈值（Q2）

**综合相似度** `score = 0.4 × nicknameSim + 0.6 × avatarSim`

| 信号 | 算法 | 说明 |
|------|------|------|
| `nicknameSim` | 归一化后 Levenshtein 相似度，或等价 fuzzy ratio | 去空格、统一大小写 |
| `avatarSim` | 头像 URL 相同 → 1.0；否则下载后 embedding cosine similarity | 联调确认 DTO 含 `avatar` |

**阈值**：

| score | 行为 |
|-------|------|
| **≥ 0.85** | **自动绑定**（`bind_status=AUTO`，`match_method=FUZZY`） |
| 0.60 ~ 0.85 | 待绑定列表高亮建议（运营一键确认） |
| < 0.60 | 仅展示在待绑定列表，不自动关联 |

> **注意**：跨通道漏斗身份（公众号↔企微↔个微好友）的 AI 增强仍属 **P2**（a1f1ec21）；本节仅指 **奥创设备 ↔ OA 个微档案** 绑定。

### 3.2 冲突与约束

- 同一 `aochuang_wechat_account_id` 不可绑两个 OA 行（Q5 UK）
- 已 `MANUAL` 绑定的行，自动同步 **不覆盖** ID，仅更新快照字段（Q4）
- `isAlive=false` 的设备：仍展示于列表，默认不参与消息增量（待联调可配置）

## 4. API 与 UI（规划）

### 4.1 奥创 OpenAPI 签名（2026-06-24 · Apifox doc-8348826）

Channel-B HTTP 客户端 `AochuangApiClient` 采用 **Signature-Version 2.0**：

| Header | 值 |
|--------|-----|
| `Account-Name` | M8 `oa_aocreate_api.app_id`（主账号登录名） |
| `Signature-Version` | `2.0` |
| `Timestamp` | Unix 秒 |
| `Signature` | `sha1(requestUri + payload + timestamp + accountName + secret)` |

- GET：`payload` = 参数按调用顺序拼接 `key=urlEncode(value)&...`（`AochuangSignatureHelper.encodeQueryString`）
- `secret` = `app_secret_encrypted` 解密后密钥

**已实现端点**（相对 `api_url` 基址）：

| 方法 | URI | 用途 |
|------|-----|------|
| GET | `/api/v1/wechatAccounts` | 设备列表（pageIndex 从 0） |
| GET | `/api/v1/wechatFriends` | 好友瀑布流 |
| GET | `/api/v1/wechatMessages` | 消息瀑布流 |

> 旧 path `/accounts/{id}/wechatAccounts` 已废弃，以本表为准。

### 4.2 规划端点

| 端点 | 说明 |
|------|------|
| `GET/POST/PUT/DELETE /config/internal-collect/aocreate/accounts` | 奥创账号子表 CRUD |
| `POST .../aocreate/accounts/{id}/test-connection` | 调 wechatAccounts 预览设备数 |
| `POST /personal-wechat/sync-devices` | 全量/单账号设备同步 + 返回待绑定 |
| `POST /personal-wechat/{id}/bind-device` | 确认绑定 / 新建并绑定 |
| `POST /personal-wechat/{id}/sync-friends` | 触发好友采集 |
| `POST /personal-wechat/{id}/sync-messages` | 触发消息采集 |

个微 Tab 工具栏：**「同步个微设备」** → 绑定对话框（左：奥创设备；右：OA 档案；待绑定 Tab）。

## 6. 铁律遵守

- 全表 `tenant_id` 隔离
- 凭证 AES-256 + 响应脱敏
- 枚举 `@InDict`；跨租户 1504
- `aochuang_account_ref_id` / 个微绑定用选择器 + 后端存在性校验（1501）

---

## 7. 后果

| 正面 | 负面 |
|------|------|
| 支持多客服工作台，无需改凭证表结构 | M8 个微 Tab UI 复杂度上升 |
| 设备发现自动化，减少手工填 ID | 模糊匹配需 avatar 字段联调与 embedding 基础设施（P2 前可用 URL 相等降级） |
| 待绑定队列避免脏数据自动建档 | `oa_aocreate_api` 仍每租户一条，多账号共享配额计数 |
