| 字段 | 值 |

---

## 1. 背景

原 PRD FR-M4-006 个微数据项仅列 `phone_id`（`<PhoneSelect />`，关联公司手机资产 `oa_phone`）。运营侧实际需要在个微账号上记录**联系用手机号**，与设备资产解耦；前端 `PersonalAccountManage.vue` 已预留 `contactPhone` 输入框，但后端未持久化。

---

## 2. 决策

### 2.1 新增字段

在 `oa_personal_wechat_account` 表新增：

| 列 | 类型 | 说明 |
|----|------|------|
| `contact_phone` | `VARCHAR(20) NULL` | 个微联系手机号，**明文存储** |

### 2.2 与 `phone_id` 的关系

- **保留** `phone_id` 列（可选），不删除、不强制废弃；现有 seed / 三方关联逻辑不受影响。
- **个微 CRUD 表单**以 `contact_phone` 手动输入为主；不再要求通过 `<PhoneSelect />` 绑定公司手机资产。
- `phone_id` 若未来有采集/奥创同步场景可继续写入；与 `contact_phone` 互不替代。

### 2.3 安全与校验

- **不 AES 加密**：属运营联系信息，非身份证/ICCID 类敏感凭证；与实名人 `phone` 脱敏策略不同，列表/详情**明文展示**。
- **校验**：可选；若填写则须匹配 `^1[3-9]\d{9}$`（11 位大陆手机号）。

### 2.4 ORM 映射

- `PersonalWechatAccountDO.contactPhone` 使用 `@TableField("contact_phone")` 显式映射 snake_case 列名，避免 MyBatis-Plus 默认驼峰与库列不一致导致写入失败。

### 2.5 API

- `PersonalWechatCreateReq` / `PersonalWechatUpdateReq` / `PersonalWechatRespVO` 增加 `contactPhone`。
- 前端 create/update 须携带 `contactPhone`。

## 4. 备选方案（未采纳）

| 方案 | 原因 |
|------|------|
| 复用 `phone_id` + PhoneSelect | 联系号与设备资产语义不同，运营需独立维护 |
| `contact_phone` AES 加密 | 非凭证类字段，过度加密增加查询/展示成本 |
