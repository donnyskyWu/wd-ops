# ADR-007 · M2 知识库"最小可用实现"与遗留债务

> **状态**：提议（2026-06-09）
> **触发**：P-GATE-UNMOCK-R 反思扫描发现 M2 知识库 spec 缺位 + 4 个端点 404（S-R1 P0-1）
> **决策**：S-R1 按"前端已定义契约 + 最小后端实现"修复 P0-1/P0-2/P0-3；**4 项妥协**作为技术债务入 S-R2

---

## 背景

M2 知识库模块（M2-知识库-KnowledgeBase）后端**长期未完工**：
- `KnowledgeController` 只有 `list` + `create` 2 个方法
- 前端 `src/api/knowledge.ts` 调 6 个端点（list/detail/create/update/delete/like）
- URL 缺 `/oa/` 段，前端打 `/admin-api/knowledge/...` 实际后端路径 `/admin-api/oa/knowledge/...`
- `docs/engineering/API-M*.md` **无 API-M2 文档**
- `oa_knowledge_base` 表无 `viewCount/likeCount/isLiked` 等统计字段

**P-GATE-UNMOCK-R S-R1 必须修 4 个 P0**（用户操作必失败），但**没有 spec**。

---

## 决策

### D1：URL 前缀统一（已修）

前端 6 个 URL 全部从 `/knowledge/...` 改为 `/oa/knowledge/...`，与后端 `@RequestMapping("/admin-api/oa/knowledge")` 对齐。

### D2：补 4 个 Controller 方法（已修）

| 端点 | 方法 | 实现 |
|------|------|------|
| `GET /oa/knowledge/{id}` | `getById` | `service.getById` |
| `PUT /oa/knowledge/update` | `update` | `service.update` + DTO 校验 |
| `DELETE /oa/knowledge/delete?id={id}` | `delete` | `service.delete` + 租户校验 |
| `POST /oa/knowledge/like` | `like` | `service.toggleLike` + action 校验 |

### D3：4 项妥协（技术债务 → S-R2）

| 字段 | S-R1 行为 | 期望（S-R2） |
|------|----------|--------------|
| **`tags`** | DB 存 `String`（逗号分隔），API 层加 `parseTags/join` 转 `string[]` | 改 `oa_knowledge_base.tags` 为 `JSON` 类型 / 加 `oa_knowledge_tag` 关联表 |
| **`viewCount`** | 固定返 0 | 加 `oa_knowledge_view` 关联表（用户+知识+时间） |
| **`likeCount`** | 前端本地 state，**不持久化** | 加 `oa_knowledge_like`（userId+knowledgeId unique）表 |
| **`isLiked`** | 前端本地 state，**不持久化** | 同上，查询时按当前 userId 关联判断 |
| **`creatorName`** | 复用 DO.creator（存 username） | 联 `sys_user` 查 nickname |

### D4：Spec 草案（已产出）

`docs/engineering/API-M2-知识库.md` 作为 S-R1 输出，标注"待 Phase 2 完善"。

---

## 依据

1. **P-GATE-DICT** 已建立"spec 漏字段 → DTO 白名单"原则（M1 OpsUserVO 字段对齐）
2. **P-GATE-UNMOCK** S-B 模式"按前端已定义契约 + 最小后端实现"（IpGroup / AuthorDashboard 已用）
3. **避免蔓延**：M2 spec 缺位是历史债务，不应在 S-R1 一次性补全 spec + 实现 + 扩展统计表

---

## 后果

### 收益
- ✅ 4 个 P0 全部修复（用户操作不再 404）
- ✅ 知识库功能可用（CRUD + 收藏 5 个端点全部跑通）
- ✅ 不阻塞 S-R1 整体 Gate 验收

### 债务（必须 S-R2 解决）
- 🟡 `tags` 类型转换在前端做（脆弱，DB 不应存 String）
- 🟡 `likeCount` 刷新丢（用户体验差，UI 数字会重置）
- 🟡 `isLiked` 跨页面/刷新丢（用户点收藏后再看，状态没了）
- 🟡 `viewCount` 永远 0（报表不准）

### 风险
- 若 S-R2 推迟，**用户投诉概率高**（"我收藏的知识怎么没了"）
