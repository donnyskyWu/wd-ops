---

## 1. 问题

v9.1 新增了"数据补录"功能（`## 5.6` 内部内容分析）。运营者对缺失数据可手工补录（API_EXCEPTION/ACCOUNT_BANNED/OFFLINE_PRACTICE/OTHER）。

当补录数据与 API 采集数据**冲突**时（即同一作品同一日期既存在 API 数据，又存在补录数据），应该：
- A. 以 API 数据为准
- B. 以补录数据为准
- C. 二者各算各的，统计时合并

---

## 2. 决策

**采用方案 A：API 数据优先，补录仅作为补丁。**

具体规则：
1. `oa_content_daily` 表每条记录带 `data_source` 字段（`API` / `IMPORT`）
2. **API 优先**：同 `(content_id, stat_date)` 已有 API 数据时，**保留 API 字段**，补录数据**不覆盖** API 字段
3. 补录数据**仅补 API 缺失的字段**（如 API 缺 `comment_count` 而补录有，则用补录的 `comment_count`）
4. 统计时**两类数据都计入**，但显示时区分来源

---

## 3. 理由

- **API 是真实数据源**：API 返回的数据来自平台官方，最接近真实情况
- **补录是补丁**：补录是因 API 异常/封号/线下等原因的临时方案，不应"覆盖"真实数据
- **可追溯性**：保留 `import_record_id` 字段，反向追溯哪些数据是补录
- **数据准确性**：避免"补录数据覆盖 API"导致数据失真

---

## 4. 合并逻辑（伪代码）

```python
def merge_content_daily(content_id, stat_date, import_data):
    api_record = db.find_content_daily(content_id, stat_date, data_source='API')
    import_record = db.find_content_daily(content_id, stat_date, data_source='IMPORT')
    
    if not api_record:
        # API 无数据：直接使用补录
        insert_or_update(content_id, stat_date, data_source='IMPORT', **import_data)
    else:
        # API 有数据：仅补缺失字段
        merged = {}
        for field in ['read_count', 'like_count', 'comment_count', 'forward_count', 'follower_change']:
            if api_record[field] is not None and api_record[field] != 0:
                merged[field] = api_record[field]  # API 优先
            elif import_data.get(field) is not None:
                merged[field] = import_data[field]  # 补录兜底
            else:
                merged[field] = 0
        merged['data_source'] = 'API'  # 标记主来源
        merged['import_record_id'] = import_data.get('import_record_id')
        update(content_id, stat_date, **merged)
```

---

## 5. 异常处理

| 情况 | 处理 |
|------|------|
| API 数据比补录晚到（如先补录后 API 数据恢复） | API 数据**覆盖**补录（数据源更权威） |
| 补录数据比 API 数据晚到（补录审核通过时） | 按上述合并逻辑（API 优先） |
| 同一日期已有"已通过"补录，禁止再次补录 | 错误码 1303 |

---

## 6. 替代方案

### 方案 B：补录优先

- 优点：补录是人工确认，可能比 API 更准确
- 缺点：覆盖真实数据，且无"数据源"标识
- 不采用 ❌

### 方案 C：双轨制

- 优点：数据互不干扰
- 缺点：统计逻辑复杂（要区分"API 营收"和"补录营收"）
- 不采用 ❌

---

## 7. 影响

- 数据分析师查看统计时：补录数据**不显眼**，但可通过 `data_source` 字段筛选
- 运营管理者审核补录时：看到的是"补充 API 缺失"而非"替换 API"
- 数据准确性：保留 API 主数据，补录仅兜底
