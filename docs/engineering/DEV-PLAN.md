## 4. 执行方法规范

### 4.1 AI 实现的"五段式 Prompt"

每个 Slice 实施时，**严格按以下 5 段提示**给 AI：

#### 第 1 段：上下文

```
你是资深 Java 工程师，负责 yudao-module-oa 子模块的 [模块名]。

技术栈：Java 17 + Spring Boot 3 + MyBatis Plus 3.5 + MySQL 8
架构：芋道 yudao 二次开发
必读：docs/engineering/TECH-CONSTRAINTS.md
必读：docs/engineering/GLOBAL-CONVENTIONS.md
```

#### 第 2 段：任务定义

```
【任务】实现 [Slice ID] - [Slice 标题]
【关联 AC】[AC ID] - [AC 标题]
【关联文档】
- PRD：[docs/product/PRD-M{N}.md] § [章节]
- UX：[docs/product/UX-M{N}.md] § [章节]
- API：[docs/engineering/API-M{N}.md] § [章节]
- STATE：[docs/engineering/STATE-M{N}.md] § [章节]
- SLICES：[docs/delivery/SLICES-M{N}.md] § [章节]
```

#### 第 3 段：约束清单

```
【硬性约束】
1. 严格按 API 文档定义字段，不增不减
2. 字典字段必须用 @InDict("dict_xxx")
3. 强关联字段必须用对应 <XxxSelect /> 组件
4. 多租户隔离：所有表带 tenant_id
5. 敏感字段 AES-256 加密（idCard/phone/cookie/apiKey）
6. 错误码：1500-1504 统一语义
7. 审计日志：所有 CUD 操作记录 sys_audit_log
8. 不修改 pom.xml/application.yml（除非 Slice 计划明确）
```

#### 第 4 段：实现要求

```
【输出】
1. 完整 Java 代码（DO / DTO / Service / Controller / Mapper）
2. MyBatis XML（如需复杂 SQL）
3. 单元测试（JUnit 5 + Mockito，覆盖率 ≥ 80%）
4. 数据库 DDL（CREATE TABLE / INDEX）
5. 状态机配置（如涉及状态机）

【代码风格】
- 类：大驼峰
- 方法：小驼峰，动词开头
- 表名：oa_ 前缀
- API 路径：kebab-case
- 中文注释解释"为什么"，不解释"做了什么"
```

#### 第 5 段：自检清单

```
【自检】
- [ ] 字段名与 API 文档 100% 一致
- [ ] 所有字典字段有 @InDict
- [ ] 所有强关联字段使用选择器（前端） + 校验（后端）
- [ ] tenantId 已添加
- [ ] 加密字段有 AES-256 处理
- [ ] 错误码 1500-1504 使用正确
- [ ] 审计日志已记录 CUD
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 状态机转移定义完整（如适用）
- [ ] F+P+E 用例至少各 3 条
```

### 4.2 "阻塞问题"机制

当 AI 实施时遇到**文档不清晰**：

**禁止**：
- ❌ 用行业惯例私自推断
- ❌ 跳过不提
- ❌ 改文档来迁就实现

**必须**：
- ✅ 输出"阻塞问题清单"（格式见下）
- ✅ 暂停实现，等产品/技术澄清
- ✅ 澄清后**先更新文档，再继续实现**

**阻塞问题清单格式**：

```markdown
## 阻塞问题清单 - [Slice ID]

| # | 问题 | 影响 | 建议 |
|---|------|------|------|
| 1 | [API 文档未明确 X 默认值] | 字段处理 | 建议 X=false |
| 2 | [STATE 未定义 Y 状态] | 状态机 | 建议 Y=ENABLED |
| 3 | ... | ... | ... |

等待：[产品] 确认 → [更新文档] → [继续实现]
```

### 4.3 实施节奏（每个 Slice）

```
Day 1: 阅读 5 文档 + 输出阻塞问题清单
Day 1: 澄清阻塞问题 + 更新文档
Day 2: AI 生成代码（DO / DTO / Mapper）
Day 3: AI 生成代码（Service / Controller / 状态机）
Day 4: AI 生成单元测试 + DDL
Day 5: 自检 + 提交 PR + 审查
```

## 5. 质量门控与退出标准

### 5.1 Slice 退出标准（DoD）

每个 Slice 完成**必须**满足：

| 项 | 标准 | 验证方法 |
|----|------|---------|
| 文档 | 阻塞问题已澄清 | PR 中无 PENDING |
| 字段 | 与 API 文档 100% 一致 | 字段对比脚本 |
| 字典 | 100% 使用 `@InDict` | 静态扫描 |
| 选择器 | 100% 使用对应组件 | 前端代码审查 |
| 多租户 | 100% 带 `tenant_id` | DDL 审查 |
| 加密 | 100% 敏感字段加密 | 代码审查 |
| 错误码 | 100% 使用 1500-1504 | 静态扫描 |
| 审计 | 100% CUD 记录 | 代码审查 |
| 测试 | 覆盖率 ≥ 80% | JaCoCo |
| F 用例 | 100% 通过 | JUnit |
| P 用例 | 100% 通过 | JUnit |
| E 用例 | 100% 通过 | JUnit |
| 审查 | 1 个审查者通过 | GitHub PR |

### 5.2 迭代退出标准（DoR）

每个迭代结束**必须**满足：

| 项 | 标准 |
|----|------|
| 所有 Slice DoD | 100% |
| Pilot 抽检 | 1 个/迭代，通过 |
| 集成测试 | 跨模块接口 100% 通过 |
| 性能 | 列表 ≤ 1.5s |
| 文档 | 与代码同步更新 |
| ADR | 任何决策开 ADR |

### 5.3 阶段退出标准

| 阶段 | 退出标准 |
|------|---------|
| 阶段 0 | HelloWorld 全流程跑通 |
| 阶段 1 | P0 模块上线 + Pilot 1 通过 |
| 阶段 2 | P1 模块上线 + 业务主流程跑通 |

## 6. 代码审查清单

### 6.1 5 大铁律（🔴 必查）

| # | 铁律 | 审查项 | 不通过示例 |
|---|------|--------|-----------|
| 1 | 强关联选择器 | 5 类字段是否用选择器 | `<el-input v-model="realnameId">` ❌ |
| 2 | 数据字典 | 字典字段是否用 @InDict | 字典字段用 `String` 无注解 ❌ |
| 3 | 多租户 | 所有表/查询带 tenant_id | `WHERE id=?` 而非 `WHERE id=? AND tenant_id=?` ❌ |
| 4 | AES-256 加密 | 敏感字段是否加密 | `String idCard` 明文落库 ❌ |
| 5 | 错误码 1500-1504 | 业务异常用统一码 | `throw new RuntimeException("...")` ❌ |

### 6.2 9 项检查（🟡 推荐）

| # | 检查项 | 工具 |
|---|--------|------|
| 1 | 字段名与 API 文档一致 | diff 脚本 |
| 2 | 中文注释解释"为什么" | 代码审查 |
| 3 | 测试覆盖率 ≥ 80% | JaCoCo |
| 4 | 状态机转移定义完整 | STATE 文档对比 |
| 5 | 审计日志记录 CUD | AOP 验证 |
| 6 | 权限标识符合规范 | `@PreAuthorize` 检查 |
| 7 | 不修改公共组件（除 ADR 授权）| git diff |
| 8 | 不修改 pom.xml（除明确依赖）| git diff |
| 9 | Commit message 格式 | `feat(scope): subject` |

## 7. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| AI 推断过多 | 中 | 中 | 阻塞问题清单 + Pilot 抽检 |
| 文档与代码漂移 | 高 | 高 | 字段对比脚本 + PR 审查 |
| 性能不达标 | 中 | 中 | 阶段 0 性能基线 + 阶段 4 压测 |
| 多租户漏洞 | 低 | 高 | `@TenantId` 注解 + AOP 强制 |
| 加密失效 | 低 | 高 | 单元测试覆盖 + 安全审计 |
| 第三方依赖变更 | 中 | 中 | Lock 文件 + ADR 记录 |
| 团队不熟悉 AI 工作流 | 高 | 中 | 培训 + Pair 工作 |

## 9. 总结

| 维度 | 数据 |
|------|------|
| 阶段数 | 5（0-4） |
| 迭代数 | 8（S0-S8） |
| 总工期 | **12 周（3 个月）** |
| 团队规模 | 7 人 |
| 文档基础 | 79 文档 + 7 Pilot |
| 质量门控 | Slice/迭代/阶段 3 级 |
| 风险等级 | 中（有完整应对）|

**核心原则**：
1. ✅ **文档先行** —— 文档不全不写代码
2. ✅ **AI 加速** —— 但**不替代**人
3. ✅ **铁律必查** —— 5 大铁律 + 9 项检查
4. ✅ **阻塞即停** —— 不推断、不偷懒
5. ✅ **Pilot 验证** —— 每迭代抽检
