## 二、关键发现（静态分析）

### 2.1 整体问题

**19 页中有 18 页是"前端纯静态骨架 + 后端完整实现"脱节**：
- 后端 6 个 Controller 已就位（Metric 47 / Funnel 43 / CustomQuery 44 / Report 179 / FinanceRoi 54 / AccountCost 52 / DashboardConfig 34 / Dashboard 29）
- 前端 API 客户端（metric.ts/funnel.ts/report.ts/finance.ts）**写了但 0 页面调用**
- 19 个页面是硬编码数组 + setTimeout 假装 API + Math.random 假图

### 2.2 Wechat / Industry 2 页是 P0 spec gap

**后端无 Wechat/Industry controller**：
- 缺 `WechatDataController` + `oa_wechat_daily` 表
- 缺 `IndustryDataController` + `oa_industry_bench` 表
- 这 2 页**只能做占位**（标 Phase 2 规划）

## 七、反思与方法论沉淀

### 7.1 S-R2 复盘教训再次验证

> "**mock 改 real 是表层，aggregation 才是根**"（S-R9 反思）

本次 S-R15 触发了更严重一层的：**前端根本没接 API，连 mock-to-real 都没做**。这是"开发-集成"脱节的极端表现。

### 7.2 4 字段对照表升级

S-R11~S-R14 沉淀的 4 字段对照表升级为 5 字段：

| 字段 | 含义 | 应用 |
|------|------|------|
| 1. 前端 import ↔ API export | 端点存在性 | L-α 全 0 命中 |
| 2. API URL ↔ Controller @RequestMapping | 路径对齐 | 100% 命中 |
| 3. Controller @RequestParam ↔ Frontend params | 参数对齐 | 大多命中 |
| 4. Frontend type ↔ Backend VO | 类型对齐 | 漏斗/Metric/CustomQuery 多处 |
| 5. **后端 Controller 是否存在** | 端点存在性 | L-α.4 Wechat/Industry 命中 |

### 7.3 "后端完整 + 前端静态"是新型风险

S-R2 修过 4 个 mock 改 real；S-R9 修过 9 个 KPI 聚合；S-R15 修过 19 页**整体未接 API**。
**这是 S0-S7 Gate 阶段被"端到端 IT 通过"掩盖的盲区**——后端 IT 测 controller，前端 IT 测路由，但**没人测"前端实际调用后端"**。

**改进建议**：
- 新增"前端 Web 自动化冒烟 IT"（Playwright 走查每个路由 + 截图）
- 引入 lint 规则：禁止 `<script setup>` 中出现 `setTimeout(.*resolve)` / `Math.random` 在 view 页面
