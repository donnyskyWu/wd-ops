## 6. 反思

### 6.1 发现的 spec/数据结构问题（4 项）

1. **API-M1-运营管理.md §4.5 内容趋势** 端点**没写** —— 后端实现跟前端需要都对齐 spec，但 spec 缺这段，导致实现和 UI 都是凭推断（前端先 mock、后端没做）。**spec 必须先于代码**。
2. **API-M1-运营管理.md §4.3 账号分析** 参数 `platform` vs `platformType`：后端 controller 用 `platform`（按业务约定写），但前端 TS 类型 `account-analysis.ts` 写 `platformType` —— **type 跟 controller 不一致**就是 type drift 隐患。
3. **`oa_content_daily` 表 schema 跟 `TenantBaseDO` 不匹配**（无 `updater/update_time`）：TenantBaseDO 是框架默认继承，但分析统计表属于只读 seed 写入场景，不需要 updater 列。**DO 继承策略**应在 DDL 设计时就明确。
4. **粉丝分析 UI 没遵循**"导出按钮与查询同行"——可能 spec §FR-M1-004 没明说，但用户走查反馈就是需求。

### 6.2 教训

- **不要偷懒** mock 一个图、mock 一个数：mock 数据是 S-R1 重点整治对象，违反就是回退。
- **加新端点 = spec + DO + service + controller + IT + 前端调用** 6 步缺一不可。本 slice 一开始忘了 DO 表 schema 检查（TenantBaseDO 继承），编译过了但运行时 500。
- **前端 type 跟后端 controller 必须 1:1 对齐**：参数名、字段类型、optional/required。建议用 OpenAPI generator 自动生成 TS type 而不是手写。
- **任何新端点必须立即补 IT**：本 slice 加的 `/internal-content/{id}/trend` 没先写 IT 跑通就上线 → 2 个 bug（List import + DO schema）要到 mvn test 才暴露。
