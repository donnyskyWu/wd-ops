## 2. 真因

### 2.1 后端 `AuthorServiceImpl.dashboard()` 是 stub

`yudao-server/yudao-module-oa/src/main/java/cn/iocoder/yudao/module/oa/service/author/AuthorServiceImpl.java:131-149`

```java
public AuthorDashboardVO dashboard(Long id) {
    AuthorDO author = requireAuthor(id);
    AuthorDashboardVO vo = new AuthorDashboardVO();
    vo.setAuthorId(author.getId());
    vo.setAuthorName(author.getAuthorName());
    // ... 只设了 IP 组名
    return vo;  // KPI 全部走 VO 默认 0
}
```

**问题**：
- 未注入 `FollowerDailyMapper` / `ContentMapper`
- 未查任何 KPI 数据
- 业务方在 S-R1 / S-R2 都未发现，因 S-R1/S-R2 走的是列表 / 新建 / 编辑，无看板回归

### 2.2 前端 `AuthorDashboard.vue` 趋势图 hardcoded `Math.random()`

`ops-platform-ui-vue/src/views/operations/AuthorDashboard.vue:213, 229, 230, 231`

```javascript
data: Array.from({ length: 30 }, (_, i) => 1200000 + i * 3500 + Math.random() * 2000)
```

**问题**：
- 即便后端填了 `followerTrend` 也会被前端覆盖（未从 `d.followerTrend` 读）
- 内容图 3 序列 × 30 天 × random 永远不变（`trend`/`anchor` tab 都受影响）

### 2.3 PRD 路径与代码路由不一致

`docs/product/PRD-M1-运营管理.md:378-384` 写：
- P-M1-001 `/ops/ip-group`
- P-M1-002 `/ops/author`
- P-M1-004 `/ops/follower-analysis`
- P-M1-005 `/ops/content-analysis`
- P-M1-006 `/ops/internal-content`
- P-M1-007 `/ops/productivity`

`ops-platform-ui-vue/src/router/index.ts` 实际是 `/ip-group`, `/author`, `/fans-analysis`, `/works-analysis`, `/internal-content`, `/efficiency`。
**S-R3 已修 P-M1-003（账号分析）**，其他 6 页留 TODO。**S-R4 一并修**。

## 5. 反思 + TODO

### 5.1 发现的 spec / 数据结构问题

1. **spec ↔ schema drift**：spec §4.2.3 §5 说"作者数据自动从 `oa_content_daily / oa_follower_daily / oa_live_*` 聚合"，但**实际 schema**：
   - `oa_content` / `oa_follower_daily` 都**只有 `accountId`**，无 `authorId` 关联字段
   - `oa_account` 也无 `authorId`
   - **唯一作者→账号的关联是 `oa_author.primary_account_id`（1:1）**
   - **意味着一个作者只能统计其主推号数据，**无法**反映作者多号数据**
   - S-R4 临时方案：按主推号聚合（spec §4.2.5 §"一个作者可关联一个主推号"）
   - **TODO S-R5**：产品确认是否补 `oa_content.author_id` / `oa_account.author_id` 字段（或建中间表 `oa_account_author`）

2. **`oa_live_*` 表无 authorId 关联**（甚至可能不存在）
   - S-R4 `liveStats` 留 0（无错，但与 spec §4.2.3 §5 "直播时长" 列冲突）
   - **TODO S-R5**：补 `oa_live_session.author_id` 后再算

3. **`oa_order_attribution` 无 authorId 关联**
   - S-R4 `orderAttribution.roi` 留 0
   - **TODO S-R5**：补 `oa_order_attribution.author_id` 后再算

4. **seed 数据 CJK 字符被截断**
   - "SEED-作者张三" 列表显示为 "SEED-???"（乱码）
   - 看 V18 SQL 文件，INSERT 时字符串含中文，本地 PowerShell 文件编码问题
   - 不影响功能（API 返的是真实字符串）
   - **TODO S-R5**：重写 seed 文件用 UTF-8 编码

### 5.2 IT 覆盖不足（再次踩 S-R3 反模式 #6）

- 改 `dashboard()` 时**没第一时间加 IT**（后补）
- 修前端时也未加 IT（前端 IT 体系尚未建立）
- **TODO**：下次走查，**改 service 即加 IT**（先 IT 写死期望 → Red → 实现 → Green → 跑全量）

### 5.3 反模式（已沉淀到方法论）

- 走查前**没读完整 PRD**（跳过了 §P-M1-002-D 看板页 spec 描述）
- 应**先查后端**（不调前端就猜），但走查顺序是"前端先看 → 后端 stub → 修后端"，**反向救火**
- 建议下次：**先 grep 后端 controller / service 实现是否 stub，再去前端**（30 秒能避免 5 分钟走查）
