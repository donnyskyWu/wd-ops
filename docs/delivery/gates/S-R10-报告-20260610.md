## 二、复测新发现 2 项（S-R9 漏改）

| ID | 级别 | 问题 | 现象 |
|----|------|------|------|
| **B10** | P1 | **mini-chart 展开时不渲染 canvas** —— S-R9 报告说"mini-chart 基于 trend 数据"，实际**根本没接 trend 接口** | `renderMiniChart(el, [], {})` 传空数据 →0 canvas |

```typescript
})
```

→ 这是 **S-R9 走查报告自我审查不严** 的遗留。

**移除 onMounted 无差别渲染**：
```typescript

```

**新增按需展开逻辑**：
```typescript
const expandedUserIds = new Set<number>()
const handleExpand = async (row, expandedRows) => {
  const userId = row.userId
  if (expandedRows.find(r => r.userId === userId) && !expandedUserIds.has(userId)) {
    // 展开时按需 fetch detail + render
    expandedUserIds.add(userId)
    const detail = await getProductivityDetail(userId, {})
    const trend = detail.trend || []
    await nextTick()
    const we = wechatChartRefs[userId]
    const ve = videoChartRefs[userId]
    if (we) renderMiniChart(we, trend, wechatCharts, userId)
    if (ve) renderMiniChart(ve, trend, videoCharts, userId)
  } else if (!expandedRows.find(r => r.userId === userId)) {
    // 折叠时清理 echarts
    expandedUserIds.delete(userId)
    if (wechatCharts[userId]) { wechatCharts[userId].dispose(); delete wechatCharts[userId] }
    if (videoCharts[userId]) { videoCharts[userId].dispose(); delete videoCharts[userId] }
  }
}
```

**给两个 `<el-table>` 加 `@expand-change="handleExpand"`**（公众号 tab + 短视频 tab）

**renderMiniChart 增强空数据态**：
```typescript
if (data.length === 0) {

}
```

## 五、反思

**S-R9 走查报告里写"修复完成"实际有空头支票**（报告自我审查不严）：
- "mini-chart 改为基于 trend 数据" → 实际是 `renderMiniChart(we, [], {})` 传空数组
- "onMounted 渲染" → 实际是渲染空 chart 占位（无意义）

**走查方法论升级**：
1. **走查时不能只看 API/UI 是否工作，还要"数据真的填进去了吗"** —— 一个空 chart 容器 vs 一个有数据的 chart，视觉上可能都是空白
2. **走查报告里"已修复"必须配真实验证**（截图/canvas 数/数据样本），不能是"代码改了"的文字承诺
3. **复测的价值** —— 用户再次走查同一页暴露了 S-R9 漏改的 2 项

后续走查计划：
- 走查报告 §三.3.2 表格里每个"已修"项必须配**实证编号**（如 `tests run: N`、截图 `path/to/file`）
- 走查报告 §四.4 浏览器复测表加 **canvas 渲染数 / 数据样本数** 列
