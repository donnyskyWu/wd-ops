## 反思

1. **MyBatis-Plus `.ge(boolean, col, val)` 的 val 提前求值坑**：condition 在前看似有 short-circuit 效果，**实际 val 是 lambda 外的局部变量，先于 maybeDo() 求值**——`null.atStartOfDay()` NPE。教训：所有 lambda wrapper 写法必须拆 if 分支显式调，绝不依赖"boolean 开关"的"短路"语义
2. **schema 字段 vs UI 字段要先校准**：B4 教训——前端 importType 控件是基于"页面有补录功能"的语义联想，但实际表 schema 不支持。**走查时必须查 schema 确认每个筛选项背后有真实字段**
3. **Spring 忽略未声明 param 是隐性 bug 源**：前端发了 N 个参数，后端只声明 M 个，**M < N 时 Spring 静默忽略多余 param**，无任何 warn 提示。**IT 必须断言"未声明的 param 不会破坏请求"**——本次 IT `listFiltersCombined` 验证了这点
4. **spec 隐含 vs 显式**：S-R5 走查发现 Bug4 时 spec §4.6.1 只描述"按平台 Tab 展示"，**未明列筛选项**——这是 spec 隐含。**走查产物（IT + 文档）反向补全 spec 隐含**，比"死等 spec 写明"更高效
5. **S-R3 漏改 pageNo/size**：InternalContent list S-R3 阶段就漏改（`S-R3-报告` 没列此文件），S-R7 顺手修——教训：**S-R3 阶段全模块 grep `pageNo` 一次性改完**比分散在 S-R5/S-R6/S-R7 改更彻底
