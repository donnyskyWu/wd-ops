## 二、关键发现（静态分析）

### 2.1 整体问题

**M3 绩效 8 页全部为"前端纯静态骨架 + 后端完整实现"脱节**：
- 4 个 controller（PerfTemplate / PerfRecord / PerfResult / OrderAttribution）完整实现 17 个端点
- `api/perfTemplate.ts`、`api/perfRecord.ts`、`api/perfResult.ts`、`api/orderAttribution.ts` 均已封装
- 但 8 个 Vue 页面**零调用**，全部用 `setTimeout + 硬编码 mock` 渲染
- 4 字段对照（`pageNo→pageNum` 错位）：M3 后端用 `pageNum`，前端用 `pageNo`（与 L-α 报告一致）

**M4 内部 10 页接入程度分化**：
- 7 页（CompanyManage / RealnameManage / RealnameDetail / PhoneManage / SimcardManage / PersonalAccountManage / TripleRelManage）已接 API
- 2 页（CompanyDetail / PlatformAccountDetail）前端仍走 mock，未接 API
- 1 页（InternalAccountManage）前端已接 `platform-account` 模块（实测 OK，但有命名空间重复）
- 1 页（SimCardLinked）已接 API 但有 `setTimeout` 延迟初始化（属正常用法）
- **0 处真正的 Math.random**（之前 grep 把 `setTimeout` 误算为 mock）

### 2.2 4 字段对照（典型）

| 字段 | M3 后端 | M3 前端 | 一致 |

| 分页参数 | `pageNum` | `pageNo` | ❌ 需在 API client 层映射 |
| `isActive` | `Integer (1/0)` | `Boolean` | ⚠ 需前端映射 |

| 字段 | M4 后端 | M4 前端 | 一致 |
|------|---------|---------|------|
| 分页参数 | `pageNo` | `pageNo` | ✅ |
| `status` | 字典 `ENABLED/DISABLED` | 字符串 | ✅ |

### 2.3 Spec Gap

**0 个 P0 spec gap**——M3 / M4 18 页**全部有后端 controller 和 API 客户端**（与 L-α 报告 2 个 Wechat/Industry spec gap 形成对比）。

## 六、复盘

### 6.1 复用的模式（L-α 已沉淀）

- **3 层映射**：`searchForm.pageNo` → API `pageNum`（M3） / `pageNo`（M4），统一在前端 API client 层做参数名映射
- **`Boolean → Integer (1/0)` 映射**：`isActive: formData.isActive ? 1 : 0`
- **空数据 fallback**：当后端 series 未提供时，ECharts 用 0 数组占位而不是 `Math.random`
- **`as any` 适度使用**：跨多后端字段（`isActive` / `isEnable`）的临时映射用 `as any` 避免 TS 报错
- **错误已拦截注释**：在 catch 块用 `// 错误已拦截` 明确意图，避免误读

### 6.2 新发现

- **M3 后端统一 `pageNum`、M4 后端统一 `pageNo`**：分模块命名习惯不一致（Yudao 风格 `pageNo` vs 自定义 `pageNum`）。建议 ADR 记录
- **命名空间重叠**：`platform-account.ts` 与 `account.ts` 都有 `getPlatformAccount / updatePlatformAccount`，**前端用 alias 消歧**（`as updatePlatformAccountV2`）。长期应合并

### 6.3 待办（推动 P-GATE）

- B1 / B3（delete 端点）：M3 模板与考核记录的删除走通，需 P-GATE 后端增量
- B4 / B8（审批流 / 扩容历史）：建议新增 2 个 IT 覆盖，但需后端先提供端点
- B9（账号粉丝/作品数）：从 M0 数据采集接入，Phase 2 范围内
