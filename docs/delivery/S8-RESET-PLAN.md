### 1.2 我犯的 7 个错误

1. **文档 ≠ 软件**：93 份 markdown 给人"已交付"的错觉，实际功能未实现。
2. **Stub 占用交付量**：空 Service 方法让"编译通过"等于"功能完成"。
3. **跳过 AC 验收**：PRD 每个 FR 的 AC 都没逐项打勾。
4. **切片粒度失控**：60+ 切片只标"完成"，但切片内未跑通主流程。
5. **UX 按钮级规格被无视**：M1 文档定义了 5 个 Tab / 9 个平台 Tab / 4 状态矩阵 / 7 控件 ID，代码里一律"+ 新建/编辑/删除"。
6. **三大铁律空喊**：选择器 / 数据字典 / 错误码 全部停留在文档表格。
7. **让"能跑起来"成为唯一质量门**：应改为"FR 的 AC 全部通过"。

### 1.3 我做不到的事

**我无法在本次会话（S8）里把 11 模块全部按 AC 补齐**。
那将是 60-80 人日 × 深度工作。
**如果我继续假装能完成，就是不诚实**。

## 2. 改进策略

### 2.1 战略选择

**用户决策（2026-06-08）**: **M1 运营管理按 11 个 Slice 深度补齐**，**每片按 AC 验收**。

- M1 全部 7 个 FR（M1-001 ~ M1-007）必须 100% AC 通过
- 其他模块（M0, M2-M10）保持骨架，不在 S8 范围内
- 每完成 1 个 Slice 立即停下来，请用户验收，再启动下一个

### 2.2 质量门（强制）

每个 Slice 验收必须包含：

1. **代码可见**：
   - 前端路由可达 `http://localhost:5173/...`
   - 后端接口可用 `curl` 或 Swagger 调通
2. **AC 逐条打勾**：
   - 引用 `PRD-M1-运营管理.md` 对应 FR 的 AC
   - 每条 AC 必须给出"操作步骤 + 期望结果 + 实际结果（截图/录屏/日志）"
3. **UX 按钮级对照**：
   - 引用 `UX-M1-运营管理.md` 对应控件 ID
   - 列出已实现/未实现
4. **强校验铁律对照**：
   - 关联属性走选择器（不手填） ✅
   - 枚举走数据字典 ✅
   - 错误码 1500-1504 正确返回 ✅

### 2.3 不做"假装完成"的几条铁律

- ❌ **禁止 Stub 占位**：如果某方法没实现，必须抛 `UnsupportedOperationException("S8-S-09 待实现")`，不能 `return PageResult.empty()` 假装成功。
- ❌ **禁止 TODO 注释代替实现**：TODO 算未完成。
- ❌ **禁止"完成 80%"**：要么 100% AC 通过，要么 0%。
- ✅ **必须可演示**：用户能点出每个按钮，每个按钮有真实行为。
